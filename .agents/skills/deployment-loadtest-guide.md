# ☁️ Deployment & Load Testing Guide

이 문서는 AWS 클라우드 환경으로의 성공적인 배포와 시스템 성능 검증을 위한 가이드라인입니다.

## 1. AWS 배포 환경 (Infrastructure)
- **컨테이너화 (Docker):** 서버 A(지갑)와 서버 B(드로우)는 각각 `Dockerfile`을 작성하여 이미지로 관리한다. 환경에 구애받지 않고 배포할 수 있도록 설계한다.
- **환경 변수 관리:** AWS 배포 시 DB 엔드포인트, 외부 API 주소 등은 하드코딩하지 않고 환경 변수(Environment Variables)를 통해 주입받는다.
- **RDS 활용:** 데이터 정합성이 중요한 결제 도메인이므로, 로컬 DB가 아닌 AWS RDS(MySQL)를 사용하며, 타임아웃 및 커넥션 풀(HikariCP) 설정을 모니터링한다.

## 2. 부하 테스트 시나리오 (Load Testing with k6)
- **도구 활용:** 성능 테스트 도구로 `k6`를 사용하며, 테스트 스크립트는 프로젝트 내부 `test/load/` 폴더에 버전 관리한다.
- **점진적 부하 (Stepping Load):** 10명, 100명, 500명, 1000명 순으로 동시 접속자 수(VUs)를 늘려가며 시스템의 임계점(Breaking Point)을 찾는다.
- **성능 지표 측정:** 다음 3가지 핵심 지표를 반드시 기록한다.
    1. **TPS (Transactions Per Second):** 초당 처리 가능한 결제 건수
    2. **P95 Latency:** 전체 요청의 95%가 완료되는 데 걸리는 시간
    3. **Error Rate:** 타임아웃이나 동시성 이슈로 인한 실패율

## 3. 모니터링 및 분석 (Analysis)
- **로그 추적:** 부하 테스트 중 발생하는 병목 현상은 아키텍처 가이드에 명시된 `Trace ID`를 통해 서버 A와 서버 B의 로그를 대조하며 원인을 파악한다.
- **병목 지점 파악:** CPU 사용량, 메모리 부족, 또는 DB Lock 경합 중 어디서 병목이 발생하는지 분석하여 기록한다.
