# 🐙 Git & Commit Workflow Guide

이 문서는 프로젝트의 형상 관리 및 커밋 히스토리의 가독성을 높이기 위한 가이드라인입니다.

## 1. 커밋 메시지 컨벤션 (Commit Message Convention)
커밋 메시지는 항상 아래의 접두사(Prefix) 중 하나로 시작해야 합니다.

- `feat:` 새로운 기능 추가 (예: `feat: Wallet 엔티티 및 잔액 차감 로직 추가`)
- `fix:` 버그 또는 로직 수정 (예: `fix: 재고 차감 시 발생하는 NPE 오류 수정`)
- `refactor:` 코드 리팩토링 (기능 변화 없음)
- `test:` 테스트 코드 작성 및 수정 (예: `test: 재고 100개에 대한 1000명 동시 요청 부하 테스트 추가`)
- `chore:` 빌드 설정, 의존성, 환경 설정 등 변경
- `docs:` 문서 작성

## 2. 커밋 메시지 본문 작성 규칙
- 제목은 50자 이내로 명확하게 작성한다.
- 단순히 '무엇을(What)' 변경했는지 적는 것을 넘어, **'왜(Why)' 변경했는지** 그 의도와 기술적 근거를 본문에 간략히 기술하는 것을 권장한다.

## 3. 브랜치 전략 (Branching)
- 기능 단위로 브랜치를 분리하여 작업한다. (예: `feature/product-entity`, `test/concurrency-k6`)
- `main` 브랜치에는 테스트가 완벽하게 통과된 코드만 병합(Merge)한다.
